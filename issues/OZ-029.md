## Context

Phase 2 mutation testing, Step 4. Testing behavior when `typedef` (documented as
unsupported) is used as an ivar type and method parameter type.

## Input

```objc
/* In PXLogger.h */
typedef int PXLogSeverity;

@interface PXLogger : OZObject {
        PXLogSeverity _minLevel;
        int _messageCount;
}

- (void)setMinLevel:(PXLogSeverity)level;
- (void)logLevel:(PXLogSeverity)level message:(OZString *)msg;
```

## Observed

Transpiler exits successfully (no error, no diagnostic). Clang resolves
`PXLogSeverity` to `int` in the AST, so the transpiler sees `int` for method
parameters. But it emits the typedef name verbatim for the ivar:

```c
/* In PXLogger_ozh.h */
struct PXLogger {
        ...
        PXLogSeverity _minLevel;
        ...
};
```

The typedef definition is NOT emitted in the generated C. GCC fails:
```
PXLogger_ozh.h:11:9: error: unknown type name 'PXLogSeverity'
```

Multiple generated files that `#include "PXLogger_ozh.h"` all fail with the
same error (PXThresholdFilter, PXMovingAverageFilter, PXSystemStatus, etc.).

## Expected

Option A (preferred): Resolve typedef to underlying type in generated C:
```c
int _minLevel;
```

Option B: Emit a transpiler error:
```
oz_transpile: error: PXLogger.h: typedef 'PXLogSeverity' used as ivar type — typedef is not supported
  Use 'int' directly instead
```

## Workaround

Use `int` directly instead of typedef'd type name.

## Discussion

Interesting: Clang resolves the typedef for method parameter types (so those
compile fine), but the transpiler uses the raw type spelling for ivar types in
the struct definition. This is a partial resolution — the typedef leaks through
only in struct fields, not in function parameters.
