## Context

Using `switch` with C enum constants in a state machine (`PXDeviceManager`).
LIMITATIONS.md says switch is not supported, but the transpiler actually
generates switch/case code. The problem is that enum values are not collected.

## Input

```objc
enum PXDeviceState {
    PXDeviceStateIdle = 0,
    PXDeviceStateInitializing = 1,
    PXDeviceStateRunning = 2,
    PXDeviceStateStopped = 3
};

@implementation PXDeviceManager
- (void)start {
    switch (_state) {
        case PXDeviceStateIdle:
            /* ... */
            break;
    }
}
@end
```

## Observed

Transpiler generates switch/case code but passes enum names verbatim:
```c
switch (self->_state) {
    case PXDeviceStateIdle:
```

GCC error:
```
error: 'PXDeviceStateIdle' undeclared (first use in this function)
```

The transpiler does NOT emit a `/* TODO: SwitchStmt */` comment as
LIMITATIONS.md suggests. It generates real switch code — just without
resolving the enum values.

No transpiler diagnostic about the unsupported construct.

## Expected

Either:
1. Collect enum declarations and include them in generated C, or
2. Emit a clear diagnostic that enum values in switch cases are not resolved

## Workaround

Replaced switch/case with if/else chains, enum with `static int` constants.
See WORKAROUNDS.md entry WA-004.

## Discussion

LIMITATIONS.md says "No switch / case statements" but the transpiler does
generate them — it just fails on enum resolution. The limitation description
may be outdated.

## Resolution

Fixed in `collect.py`: added `_is_user_enum` check so user-defined enums from
the main source file are collected into `type_defs`. The enum definition is
emitted in the class header when the ivar type matches (e.g., `enum PXDeviceState`).
Updated LIMITATIONS.md — switch/case is now documented as supported with C enums.

Tests added: `test_user_enum_from_main_file`, `test_enum_ivar_type_emitted_in_header`.
