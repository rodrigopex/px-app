## Context

Phase 2 mutation testing, Step 6. Testing behavior when a class declares protocol
conformance but omits a required protocol method.

## Input

```objc
/* PXTemperatureSensor.h declares <PXSensorProtocol> which requires readValue */
/* PXTemperatureSensor.m — readValue method removed: */
@implementation PXTemperatureSensor

- (id)initWithId:(int)sensorId { ... }
- (int)sensorId { ... }
/* readValue MISSING — protocol requires it */
- (OZString *)name { ... }

@end
```

## Observed

- Transpiler: no error, no diagnostic. Exits successfully.
- GCC: compiles and links successfully.
- Runtime: QEMU hangs/crashes. The vtable entry for `readValue` on
  PXTemperatureSensor is NULL (zero-initialized). When protocol dispatch
  calls `OZ_SEND_readValue()` on a PXTemperatureSensor, it dereferences
  a NULL function pointer.

Generated vtable initialization in `oz_dispatch.c`:
```c
OZ_vtable_readValue[OZ_CLASS_PXHumiditySensor] = (OZ_fn_readValue)PXHumiditySensor_readValue;
/* No entry for PXTemperatureSensor — slot remains NULL */
```

## Expected

Transpiler should check that every class declaring protocol conformance
(`<PXSensorProtocol>`) implements all required protocol methods. Error:
```
oz_transpile: error: PXTemperatureSensor.m: class 'PXTemperatureSensor' conforms to
  protocol 'PXSensorProtocol' but does not implement required method 'readValue'
```

Alternatively, the vtable initialization could insert a stub function that prints
an error message instead of crashing via NULL pointer.

## Workaround

None needed — this is a mutation test. In production code, always implement all
protocol methods.

## Discussion

This is a critical safety issue. A missing protocol method compiles, links, and
only crashes at runtime with no diagnostic message. The transpiler has the class
model and protocol definitions — it should verify conformance.

## Verification (FAILED)

DEV re-applied mutation (removed `readValue` from PXTemperatureSensor.m).
- Transpiler: no error, no diagnostic. Exits successfully.
- Build: compiles and links without error.
- Vtable: `OZ_vtable_readValue[OZ_CLASS_PXTemperatureSensor]` is still NULL.
- Runtime: will crash on `OZ_SEND_readValue()` for PXTemperatureSensor.

The conformance check reported by QA does not appear to trigger for this case.
Possible reasons:
1. The check only runs when `readValue` is dispatched via protocol, but the
   vtable is populated based on implementations, not protocol declarations.
2. The check may require `--strict` which isn't passed at build-time
   (only configure-time).

**Status: REOPENED** — fix does not prevent the runtime crash.
