## Context

Phase 2 verification of OZ-027 fix. The original issue was that capturing blocks
were classified as warnings (invisible without --verbose) and generated an error
comment that broke GCC.

## Input

```objc
/* In main.m */
int capturedValue = 42;
[pub publishSensorId:1 value:99 timestamp:999 completion:^(int status) {
        _blockStatus = status + capturedValue;
        OZLog("  block callback: %d captured: %d\n", status, capturedValue);
}];
```

## Observed

After OZ-027 fix, the transpiler no longer emits a warning or error comment.
But it generates:

```c
int capturedValue = 42;
PXZbusPublisher_publishSensorId_value_timestamp_completion_(pub, 1, 99, 999, ((void *)0));
```

The capturing block is replaced with NULL. The variable `capturedValue` is
declared but unused (GCC warns). The block body is completely dropped. If the
method dereferences the NULL callback, it will crash at runtime.

This is the same regression pattern as OZ-035: unsupported constructs now compile
to NULL instead of failing at compile time.

## Expected

Since capturing blocks are documented as unsupported, the transpiler should
emit an error and stop:
```
oz_transpile: error: main.m: capturing blocks not supported (block captures 'capturedValue')
  Use a non-capturing block or a file-scope static variable instead.
```

## Workaround

Use file-scope `static` variable instead of local capture.

## Discussion

Same root cause as OZ-035: unsupported constructs are being silently converted to
NULL instead of producing transpiler errors.
