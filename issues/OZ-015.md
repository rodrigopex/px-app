## Context

Commit `f8abdc8` (OZ-013 fix) adds `static` qualifier to re-emitted file-scope
variables. However, it also emits `extern` declarations for these same variables
in the generated `_ozh.h` header. This causes GCC errors when the `.c` file
declares them as `static`.

All builds broken — affects PXAppConfig (`_sharedConfig`), PXLogger
(`_sharedLogger`), and any class with file-scope statics.

## Input

```objc
/* PXAppConfig.m */
static PXAppConfig *_sharedConfig;

@implementation PXAppConfig
+ (void)initialize {
    _sharedConfig = [[PXAppConfig alloc] init];
}
@end
```

## Observed

Generated `PXAppConfig_ozh.h`:
```c
extern struct PXAppConfig * _sharedConfig;  /* ← WRONG: extern in header */
```

Generated `PXAppConfig_ozm.c`:
```c
static struct PXAppConfig * _sharedConfig;  /* ← correct: static in .c */
```

GCC error:
```
error: static declaration of '_sharedConfig' follows non-static declaration
```

## Expected

File-scope `static` variables should NOT appear in the `_ozh.h` header at all.
They are private to the translation unit — `extern` makes them public, which
contradicts the `static` semantics.

## Workaround

None — all builds broken.

## Discussion

(empty)

## Resolution

Removed `extern` declarations for static variables from `_class_header_ctx` in
emit.py. File-scope `static` variables are TU-private and must not be exported
in headers.
