## Context

Discovered via px-app CHANGELOG Step 9 and OZ-007. LIMITATIONS.md states:

> No `switch` / `case` statements. The transpiler does not handle
> `SwitchStmt` or `CaseStmt` AST nodes.

This is inaccurate. The transpiler **does** generate switch/case code.
The actual limitation is that **enum values referenced in case labels
are not collected**, so they appear as undeclared identifiers in the
generated C.

DEV's OZ-007 confirms: switch/case is emitted, but `PXDeviceStateIdle`
(an enum constant) is not defined in the generated output.

## Expected

Update LIMITATIONS.md to accurately describe the real limitation:
- Switch/case is partially supported (the control flow is emitted)
- Enum declarations are not collected by the transpiler
- Case labels referencing enum constants will produce GCC errors
- Workaround: use `static int` constants with if/else, or integer literals
  in case labels

## Resolution

Already fixed in commit `8523e65` (OZ-007). LIMITATIONS.md now correctly states
that switch/case is supported with C enum types, and user-defined enums from the
main source file are collected and emitted in the class header.
