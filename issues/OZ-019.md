## Context

In ObjC, a subclass can access ivars declared in a parent class directly
(e.g., `_sampleCount` in the subclass implementation when `_sampleCount`
is declared in the superclass `@interface`). The transpiler emits
`self->_sampleCount` but the C struct for the subclass doesn't have
`_sampleCount` at the top level — it's in the embedded base struct.

## Input

```objc
@interface PXSensorBase : OZObject {
    int _sampleCount;
}
@end

@interface PXAnalogSensor : PXSensorBase {
    int _rawValue;
}
@end

@implementation PXAnalogSensor
- (int)readRaw {
    _rawValue = _rawValue + 5;
    _sampleCount = _sampleCount + 1;  /* parent ivar */
    return _rawValue;
}
@end
```

## Observed

Generated `PXAnalogSensor_ozm.c`:
```c
int PXAnalogSensor_readRaw(struct PXAnalogSensor *self)
{
    self->_rawValue = self->_rawValue + 5;
    self->_sampleCount = self->_sampleCount + 1;  /* ERROR */
    return self->_rawValue;
}
```

Generated struct layout:
```c
struct PXSensorBase {
    struct OZObject base;
    int _sensorId;
    int _sampleCount;
};

struct PXAnalogSensor {
    struct PXSensorBase base;  /* _sampleCount is in here */
    int _rawValue;
    int _calibrationOffset;
};
```

GCC error:
```
error: 'struct PXAnalogSensor' has no member named '_sampleCount'
```

## Expected

The transpiler should navigate the struct embedding chain to access parent ivars:
```c
self->base._sampleCount = self->base._sampleCount + 1;
```

For deeper hierarchies (grandparent ivars), it would be:
```c
self->base.base._grandparentIvar
```

The transpiler already knows the class hierarchy (it builds the inheritance
chain for struct embedding). It should use the same chain to resolve ivar
access for inherited fields.

## Workaround

Avoid direct parent ivar access in subclass implementations. Use accessor
methods defined in the parent class instead:
```objc
/* Instead of: _sampleCount = _sampleCount + 1; */
/* Use an accessor if available, or don't access parent ivars */
```

See WORKAROUNDS.md entry WA-006.

## Discussion

This affects any inheritance depth > 1. The existing 2-level samples
(OZObject → subclass) work because OZObject's ivars (`oz_class_id`,
`_refcount`) are accessed via macros/inline functions, not direct ivar access.
