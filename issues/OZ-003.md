## Context

Using a protocol method that returns `OZString *` via protocol dispatch
(`id<PXSensorProtocol>` variable).

## Input

```objc
@protocol PXSensorProtocol <OZProtocol>
- (OZString *)name;
- (int)sensorId;
- (int)readValue;
@end

/* In main.m: */
id<PXSensorProtocol> sensor = (id<PXSensorProtocol>)temp;
OZString *sensorName = [sensor name];
```

## Observed

Generated C uses placeholder type `struct __patched__`:
```c
struct OZString * sensorName = (struct __patched__ *)OZ_SEND_name(_oz_recv1);
```

GCC error:
```
error: initialization of 'struct OZString *' from incompatible pointer type 'struct __patched__ *'
```

The `int`-returning protocol methods (`sensorId`, `readValue`) work correctly
via `OZ_SEND_*` dispatch. Only object pointer return types are affected.

## Expected

```c
struct OZString * sensorName = (struct OZString *)OZ_SEND_name(_oz_recv1);
```

## Workaround

Call `name` on the concrete type (`PXTemperatureSensor *`) instead of through
protocol dispatch. Protocol dispatch only used for `int`-returning methods.
See WORKAROUNDS.md entry WA-002.

## Discussion

All existing protocol behavior tests only use `int`/`void` return types.
This code path (protocol method → object pointer return) appears untested.

## Resolution

Fixed in `emit.py`: protocol dispatch cast now uses the declared return type
from the AST (`ret_oz.c_type`) instead of the inferred receiver class. This
prevents `struct __patched__` from leaking when a protocol method returning
an object pointer is called from a transpiled function context.

Test added: `test_protocol_dispatch_object_return_cast` — verifies
`(struct OZString *)OZ_SEND_name(...)` is emitted without `__patched__`.
