## Context

LIMITATIONS.md does not document two constructs that generate invalid C
(discovered in px-app Step 1, see OZ-002 and OZ-004):

1. `@synthesize propName;` (without explicit ivar assignment) — generates
   struct/accessor mismatch. The workaround (`@synthesize propName = _propName;`)
   is not documented anywhere.

2. File-scope static variables with ObjC types (`static ClassName *var = nil;`)
   are emitted verbatim and produce invalid C. The workaround (omit `= nil`,
   initialize in `+initialize`) is not documented.

These are the kind of limitations a new user of the transpiler would hit
immediately when following standard ObjC patterns.

## Expected

Add entries to LIMITATIONS.md under an appropriate section (e.g., "Properties"
and "Types" or "Miscellaneous") documenting:
- `@synthesize` requires explicit ivar form
- File-scope statics with ObjC class types require special handling

## Resolution

Added to `docs/LIMITATIONS.md`:
- **Properties section**: Documents `@synthesize` requiring explicit ivar form,
  with explanation that the bare form uses the bare property name as ivar.
- **Types section**: Documents file-scope statics with ObjC types, including the
  workaround (declare without initializer, assign in `+initialize`).
