## Context

Phase 2 mutation testing, Step 5. Testing behavior when calling a method that
belongs to one class on an object of a different class.

## Input

```objc
/* In main.m */
PXDeviceManager *mgr = [[PXDeviceManager alloc] init];
[mgr start];
[mgr info:@"test"];  /* info: is a PXLogger method, not PXDeviceManager */
```

## Observed

Transpiler exits successfully (no error, no diagnostic). Generated C:
```c
PXDeviceManager_info_(mgr, _oz_str_...);
```

GCC then fails:
```
main_ozm.c:90:9: error: implicit declaration of function 'PXDeviceManager_info_';
  did you mean 'PXDeviceManager_init'?
```

- No transpiler error or diagnostic
- Transpiler blindly constructs `ClassName_methodName_()` without checking if the
  method exists on that class
- GCC error points at generated C, not source .m file
- Clang also doesn't catch this (ObjC allows sending any message to a typed receiver)

## Expected

Transpiler should warn or error when a method call targets a class that doesn't
declare that method:
```
oz_transpile: warning: main.m:63: method 'info:' not declared on class 'PXDeviceManager'
  Did you mean to call this on a PXLogger instance?
```

This would catch a very common developer mistake — calling the right method on
the wrong object.

## Workaround

None needed — GCC catches it. But the GCC error references generated C internals
(`PXDeviceManager_info_`) rather than the source ObjC.

## Discussion

This is a type-checking improvement request, not a silent-invalid-C bug. The
transpiler could check its class model to verify the method exists. Low priority
compared to silent-failure issues, but would significantly improve developer
experience since this is a very common mistake.
