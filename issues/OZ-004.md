## Context

Discovered via px-app CHANGELOG Step 1. DEV used a file-scope static variable
with an ObjC type (`static PXAppConfig *_sharedConfig = nil;`) and the
transpiler emitted it verbatim. `PXAppConfig` is not a valid C type — it needs
to be transpiled to the generated struct type. DEV worked around it by dropping
the `= nil` initializer and following the arc_demo pattern.

## Input

```objc
#import "PXAppConfig.h"

static PXAppConfig *_sharedConfig = nil;

@implementation PXAppConfig
+ (PXAppConfig *)shared {
    return _sharedConfig;
}
@end
```

## Observed

The transpiler emits the static declaration verbatim in the generated C:
```c
static PXAppConfig *_sharedConfig = nil;
```

GCC fails because:
1. `PXAppConfig` is not a C type (should be `struct PXAppConfig_impl *`)
2. `nil` is not defined in plain C (should be `NULL`)

## Expected

The transpiler should transform file-scope static declarations with ObjC types:
```c
static struct PXAppConfig_impl *_sharedConfig = NULL;
```

Or at minimum, emit a diagnostic that file-scope statics with ObjC types
are not fully transpiled.

## Workaround

Declare without initializer: `static PXAppConfig *_sharedConfig;` and
initialize in `+initialize` method. See arc_demo pattern.

## Resolution

Fixed in `collect.py`: `_extract_init_value` now handles `nil`/`NULL` initializers:
- `GNUNullExpr` → `NULL`
- `ImplicitCastExpr` with `NullToPointer` castKind → `NULL`
- `CStyleCastExpr` with `NullToPointer` castKind → `NULL`

This ensures `static PXAppConfig *_shared = nil;` is collected and emitted as
`struct PXAppConfig *_shared = NULL;` (ObjC type transformed to C struct pointer).

When a static with ObjC type has an unrecognized initializer, a diagnostic
warning is now emitted (OZ-006 improvement).

Tests added: `test_static_with_gnu_null_init`, `test_static_with_null_to_pointer_cast`,
`test_static_with_cstyle_null_cast`, `test_static_objc_type_unsupported_init_warns`.
