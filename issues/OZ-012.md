## Context

Discovered via px-app CHANGELOG "OZ-007/OZ-009 verified" entry. The OZ-007
fix correctly collects and emits enum values, but the switch/case statement
itself is not properly emitted. Only the condition expression (`self->_state;`)
appears as a bare statement — all case labels and bodies are lost.

This means the LIMITATIONS.md entry updated by OZ-010 (which now says
"switch/case supported with C enum types") is inaccurate again. Switch/case
is NOT supported — only enum collection is.

## Input

```objc
switch (_state) {
    case PXDeviceStateIdle:
        _state = PXDeviceStateInitializing;
        break;
    case PXDeviceStateRunning:
        _errorCount++;
        break;
}
```

## Observed

Generated C (per DEV's verification):
```c
self->_state;
```

The entire switch structure (cases, bodies, breaks) is dropped. Only the
condition expression remains as a bare statement.

## Expected

```c
switch (self->_state) {
    case PXDeviceStateIdle:
        self->_state = PXDeviceStateInitializing;
        break;
    case PXDeviceStateRunning:
        self->_errorCount++;
        break;
}
```

## Workaround

Use if/else chains with enum constants (WA-004). Enum values work correctly.

(empty)

## Resolution

Fixed in `emit.py`: added `_emit_switch_stmt`, `_emit_case_stmt`, `DefaultStmt`
handling in `_emit_stmt`, and `ConstantExpr` support in `_emit_expr`. Switch/case
is now fully emitted with condition, case labels, bodies, break, and default.

Test added: `test_switch_case_emitted`.
