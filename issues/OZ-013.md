## Context

When a `.m` file mixes ObjC classes with plain C code at file scope
(macro invocations, C function definitions, C struct definitions), the
transpiler strips all non-ObjC constructs from the generated output.
The `#include` directives ARE preserved (OZ-011 verified), but the C
constructs themselves are lost.

This matters for Zephyr interop where C macros (`ZBUS_CHAN_DEFINE`,
`K_THREAD_DEFINE`, etc.) and C helper functions commonly live alongside
ObjC class implementations.

## Input

```objc
/* PXZbusPublisher.m */
#import "PXZbusPublisher.h"
#include "px_zbus_defs.h"
#include <zephyr/kernel.h>

/* 1. C macro invocation — defines a zbus channel */
ZBUS_CHAN_DEFINE(px_sensor_chan,
                struct px_sensor_msg,
                NULL, NULL,
                ZBUS_OBSERVERS_EMPTY,
                ZBUS_MSG_INIT(0));

/* 2. File-scope static variable */
static int _lastPublishedValue;

/* 3. Plain C function definition */
int px_zbus_publish(int sensor_id, int value, int timestamp)
{
    struct px_sensor_msg msg;
    msg.sensor_id = sensor_id;
    msg.value = value;
    msg.timestamp = timestamp;
    int err = zbus_chan_pub(&px_sensor_chan, &msg, K_MSEC(100));
    if (err == 0) {
        _lastPublishedValue = value;
    }
    return err;
}

/* 4. Another plain C function */
int px_zbus_last_value(void)
{
    return _lastPublishedValue;
}

/* ObjC class — this part transpiles correctly */
@implementation PXZbusPublisher
- (id)init { ... }
@end
```

## Observed

Generated `PXZbusPublisher_ozm.c` contains:

```c
#include "PXZbusPublisher_ozh.h"
/* ... auto-generated includes ... */

OZ_SLAB_DEFINE(oz_slab_PXZbusPublisher, ...);

#include "px_zbus_defs.h"      /* ← PRESERVED (OZ-011 fix) */
#include <zephyr/kernel.h>     /* ← PRESERVED */

/* MISSING: ZBUS_CHAN_DEFINE(...) macro invocation */
/* MISSING: static int _lastPublishedValue; */
/* MISSING: int px_zbus_publish(...) function definition */
/* MISSING: int px_zbus_last_value(...) function definition */

int _blockResult;               /* ← static qualifier lost */

/* ObjC methods transpiled correctly below */
struct PXZbusPublisher * PXZbusPublisher_init(...) { ... }
```

Three categories of C constructs are stripped:

### 1. C macro invocations at file scope
`ZBUS_CHAN_DEFINE(...)` is a macro that Clang expands before AST generation.
The expanded declarations (struct defs, extern vars) appear in the AST as
regular C declarations, but `_walk()` in `collect.py` only collects ObjC-specific
node types (`ObjCInterfaceDecl`, `ObjCImplementationDecl`, `ObjCMethodDecl`,
`EnumDecl`, etc.) and skips plain C declarations.

The original macro text is lost after Clang preprocessing, so the transpiler
cannot reconstruct it. The expanded form would need to be collected and emitted.

### 2. Plain C function definitions
`int px_zbus_publish(...)` is a `FunctionDecl` in the Clang AST. The transpiler's
`_walk()` does not collect `FunctionDecl` nodes from user source files. Only
`static` functions referenced from blocks/ObjC methods are partially handled.

### 3. File-scope static variables (partial)
`static int _lastPublishedValue;` — the transpiler partially collects statics
(OZ-004 fix handles `static ClassName *var = nil;`), but plain `static int`
variables not referenced from ObjC methods may be dropped. When emitted,
the `static` qualifier is sometimes lost (e.g., `_blockResult` emitted
without `static`).

## Expected

Non-ObjC top-level declarations in user `.m` files should be passed through
to the generated output. The transpiler already knows which file is the "user"
source (vs. included headers) via `last_file` tracking in `_walk()`.

Approach options:
1. **Roundtrip passthrough**: For any top-level AST node in the user source file
   that is NOT an ObjC construct, extract the original source text (using Clang's
   source location info) and emit it verbatim in the generated output.
2. **Collect FunctionDecl**: Add `FunctionDecl` handling to `_walk()`, similar to
   how `EnumDecl` was added in OZ-007. Collect the function body and emit it.
3. **Patched-source approach**: The existing `_emit_patched_source()` path already
   handles roundtrip for some constructs. Extend it to preserve non-ObjC
   declarations that appear between `@implementation` blocks.

For C macros specifically: since the macro text is lost after preprocessing,
the transpiler would need to either:
- Use Clang's `-detailed-preprocessing-record` to capture macro expansions
- Or document that C macros must live in separate `.c` files (current workaround)

## Workaround

C macro invocations and C function definitions live in a separate `.c` file
compiled directly via CMake `target_sources()`. The `.m` file only contains
ObjC class code and calls the C functions. See WORKAROUNDS.md entry WA-005.

## Discussion

The `arc_demo` sample's `static Sensor *createSensor(int v)` function IS
emitted in the generated output — this works because it's a `static` function
that returns an ObjC type and is referenced from ObjC code. The transpiler
collects it as part of the ObjC roundtrip. But plain C functions with no ObjC
type references are not collected.

The zbus_service sample (`TemperatureService.m`) uses this same pattern
successfully — it has `ZBUS_CHAN_DEFINE`, C functions, and `@implementation`
in one file. This suggests the transpiler's patched-source path may handle
this correctly for single-file projects but fail for multi-file projects
where class headers are in separate `.h` files.

## Resolution (MAINTAINER — could not reproduce)

MAINTAINER's test preserved C constructs. However, DEV retested and the issue
persists. Updated evidence from generated `PXZbusPublisher_ozm.c`:

**What IS preserved (partial success):**
- C function BODIES: `px_zbus_publish()` and `px_zbus_last_value()` appear
  at lines 51-67 of generated output — the function code survives
- Static variable NAMES: `_lastPublishedValue` and `_blockResult` appear
  (but lose `static` qualifier)

**What is STILL stripped:**
1. `#include "px_zbus_defs.h"` — quoted include stripped (struct `px_sensor_msg`
   and `zbus_chan_pub` are undeclared). This contradicts OZ-011 "could not
   reproduce" — in this specific file the include IS stripped.
2. `#include <zephyr/kernel.h>` — angle-bracket include stripped (needed for
   `k_timeout_t`, `K_MSEC`)
3. `ZBUS_CHAN_DEFINE(...)` — macro invocation stripped (channel definition lost)
4. `static` qualifier lost on file-scope variables

**Possible root cause:** The patched-source path may work differently when:
- The `.m` file has C code BEFORE `@implementation` (includes, macros, functions)
- Multiple include directives are present
- The file uses both `#import` and `#include`

The `.m` file is listed in `objz_transpile_sources()` in CMakeLists.txt.

**ROOT CAUSE FOUND (DEV 2026-03-15):**

The patched-source path works for `main.m` (no `@implementation`) but NOT for
files with `@implementation`/`@end`. Evidence:

- `main_ozm.c` line 31-32: `#include <zephyr/kernel.h>` and
  `#include "services/px_zbus_defs.h"` both preserved — main.m has no
  `@implementation`, so patched-source path is used for the entire file.

- `PXZbusPublisher_ozm.c`: NO includes preserved, NO macros, C function bodies
  survive but without their required headers. This file HAS `@implementation`,
  which triggers the template emission path for the class, and the
  pre-`@implementation` C code (includes, macros, function definitions) is lost.

The emission path diverges based on `@implementation` presence:
- Without `@implementation` → patched-source path → C code preserved
- With `@implementation` → template path for class + partial roundtrip →
  pre-implementation C code stripped
