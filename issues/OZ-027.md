## Context

Phase 2 mutation testing, Step 3. Testing severity classification for capturing
blocks (documented unsupported construct).

## Input

```objc
/* In main.m, block callback section */
int capturedValue = 42;
[pub publishSensorId:1 value:99 timestamp:999 completion:^(int status) {
        _blockStatus = status + capturedValue;
        OZLog("  block callback: %d captured: %d\n", status, capturedValue);
}];
```

## Observed

Transpiler detects the issue and shows a diagnostic:
```
oz_transpile: warning: error: capturing blocks not supported (block captures 'capturedValue'). Use a regular loop instead.
```

But it continues generating files (exits with code 0). Generated C:
```c
PXZbusPublisher_publishSensorId_value_timestamp_completion_(pub, 1, 99, 999, /* ERROR: capturing block (captures 'capturedValue') */);
```

GCC then fails:
```
main_ozm.c:126:141: error: expected expression before ')' token
```

Three issues:
1. **Severity is wrong:** Classified as warning, should be error. The generated C
   is invalid — the block placeholder comment causes a mandatory GCC failure.
2. **Confusing "warning: error:" prefix:** The message reads
   `oz_transpile: warning: error: capturing blocks...` — the inner "error:" is
   the diagnostic text, making it look like a warning wrapping an error.
3. **No source file/line:** Does not identify which .m file or line contains the
   capturing block.

## Expected

```
oz_transpile: error: main.m:96: capturing blocks not supported (block captures 'capturedValue')
  Use a non-capturing block or a file-scope static variable instead.
```

This should be `module.errors` (not `module.diagnostics`) since it always produces
invalid C. The transpiler should exit with code 1 and not generate output.

## Workaround

Use file-scope `static` variable instead of local capture (already the pattern
in the working code with `_blockStatus`).
