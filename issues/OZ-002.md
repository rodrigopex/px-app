## Context

Discovered via px-app CHANGELOG Step 1. DEV used `@synthesize propName;`
(modern ObjC default form) and the transpiler generated invalid C.
DEV classified this as "convention, not bug" and adopted `@synthesize propName = _propName;`
as a workaround. QA disagrees — the transpiler either needs to support the
modern form or emit a diagnostic rejecting it.

## Input

```objc
@interface PXAppConfig : OZObject
@property int sampleRate;
@end

@implementation PXAppConfig
@synthesize sampleRate;
@end
```

## Observed

The transpiler generates a struct field named `sampleRate` (no underscore) but
accessor code that references `self->_sampleRate` (with underscore). GCC fails
because the struct has no `_sampleRate` field.

The modern ObjC convention is that `@synthesize propName;` implicitly maps to
`@synthesize propName = _propName;` — the backing ivar gets an underscore prefix.
The transpiler appears to partially implement this (accessor uses `_propName`)
but the struct definition uses the bare name.

## Expected

Either:
1. **Support the modern form:** `@synthesize propName;` should generate both
   struct field and accessors using `_propName`, or
2. **Emit a diagnostic:** warn/error that the explicit ivar form
   `@synthesize propName = _propName;` is required, with a suggested fix

## Workaround

Use explicit form: `@synthesize propName = _propName;` (all existing samples
and tests already use this form).

(empty)

## Resolution

Fixed in `resolve.py`: `_synthesize_properties` now checks `cls.ivars` for an
existing ivar matching the bare property name before defaulting to `_propName`.
When `@synthesize propName;` is used and Clang creates an ivar named `propName`,
the accessor now correctly references `self->propName` (matching the struct field).

A diagnostic warning is emitted suggesting the explicit form
`@synthesize propName = _propName;` for clarity.

Tests added: `test_bare_synthesize_uses_existing_ivar`,
`test_bare_synthesize_accessor_matches_struct`.
