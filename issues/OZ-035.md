## Context

Phase 2 verification of OZ-025 fix. The original issue was that `@(expr)` boxed
expressions emitted `/* TODO: ObjCBoxedExpr(ParenExpr) */` which broke GCC.

## Input

```objc
/* In PXMovingAverageFilter.m */
OZNumber *r = @(_sum / _count);
return [r intValue];
```

## Observed

After OZ-025 fix, the transpiler no longer emits the TODO comment. But it now
generates:

```c
struct OZNumber * r = ((void *)0);
OZObject_release((struct OZObject *)r);
return OZNumber_intValue(r);
```

The boxed expression `@(_sum / _count)` becomes `NULL`. The code compiles and
links successfully, but `OZNumber_intValue(NULL)` will crash at runtime.

This is a regression from the original behavior: before OZ-025, the construct
caused a compile error (GCC caught the TODO comment). Now it silently compiles
but crashes at runtime — a worse outcome.

## Expected

Since boxed expressions are documented as unsupported, the transpiler should
emit an error and stop:
```
oz_transpile: error: PXMovingAverageFilter.m: boxed expression @(...) is not supported
```

Or if the intent was to support boxed expressions, then `@(_sum / _count)` should
generate `[[OZNumber alloc] initWithInt:(_sum / _count)]`.

## Workaround

Use `int` directly instead of boxing to OZNumber.
