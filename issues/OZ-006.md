## Context

Cross-cutting issue discovered via px-app CHANGELOG Steps 1 and 3. Problems
DEV encountered (OZ-001 subdirectory includes, OZ-002 @synthesize mismatch,
OZ-004 file-scope static types, OZ-003 `__patched__` placeholder in protocol
dispatch) all resulted in the transpiler silently emitting invalid C. None
produced a transpiler-level warning or error. DEV only discovered the problems
when GCC failed on the generated output.

This is a usability problem: the transpiler should validate its own output or
detect unsupported constructs before emitting them, rather than pushing
all error discovery to the downstream compiler.

## Input

Any of the inputs from OZ-001, OZ-002, OZ-003, or OZ-004.

## Observed

The transpiler exits successfully (exit code 0) and produces `.c`/`.h` files
that do not compile. No warnings, no errors, no `--verbose` output indicating
a problem. The user has to read GCC errors on *generated* C and reverse-engineer
which ObjC construct caused the issue.

## Expected

For constructs the transpiler cannot fully handle, it should either:
1. **Emit a warning** naming the source `.m` file, line number, and construct
   (e.g., "warning: @synthesize without explicit ivar not fully supported,
   use @synthesize prop = _prop")
2. **Emit an error in `--strict` mode** for constructs that produce invalid C
3. **At minimum**, add a `/* WARNING: ... */` comment in the generated C at the
   point of the untranspiled construct, so the GCC error message has context

## Workaround

None — the user must manually diff `.m` input against generated `.c` output
to find the unsupported construct.

## Discussion

(empty)

## Resolution

Addressed as part of OZ-002 and OZ-004 fixes:

1. **`@synthesize propName;` (OZ-002)**: `resolve.py` now emits a diagnostic
   warning when bare synthesize form is detected, suggesting the explicit form.

2. **File-scope static with ObjC type (OZ-004)**: `collect.py` now emits a
   diagnostic warning when a static variable with an ObjC class type has an
   unsupported initializer, with actionable guidance ("declare without initializer
   and assign in +initialize").

Both diagnostics appear in `--verbose` output via `module.diagnostics`.
The underlying bugs that caused silent invalid C generation are now fixed
(OZ-002: struct/accessor mismatch, OZ-004: nil init not recognized).
