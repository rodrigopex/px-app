## Context

Phase 1 revealed a recurring pattern: the transpiler silently generates invalid C
for unsupported constructs. The developer only discovers problems when GCC fails
on the generated output with cryptic errors referencing transpiler internals
(`struct __patched__`, `_oz_recv0`, missing includes). The transpiler has a
diagnostic system (`module.diagnostics` for warnings, `module.errors` for fatal
errors, `--verbose`/`--strict` flags), but `oz_transpile.cmake` does not pass
either flag.

## Input

`oz_transpile.cmake` lines 129-138 â€” the transpiler invocation:

```cmake
execute_process(
    COMMAND ${CMAKE_COMMAND} -E env PYTHONPATH=${_transpile_dir}
            ${Python3_EXECUTABLE} -m oz_transpile
            --input ${_ast_files}
            --sources ${_abs_sources}
            --outdir ${_outdir}
            --root-class=${OZT_ROOT_CLASS}
            --manifest=${_manifest}
            ${_pool_flag}
    RESULT_VARIABLE _rc
)
```

Same pattern in the build-time shell script (lines 166-174). Neither invocation
passes `--verbose` or `--strict`.

## Observed

- Diagnostics appended to `module.diagnostics` (warnings) are invisible during
  `west build`. Only `module.errors` cause transpiler exit code != 0.
- Constructs like boxed expressions emit `/* TODO: ObjCBoxedExpr */` comments
  silently â€” no warning, no error. GCC then fails on the incomplete C.
- Capturing blocks produce a diagnostic (warning) that is invisible without
  `--verbose`.

## Expected

`oz_transpile.cmake` should pass `--strict` (or at minimum `--verbose`) to the
transpiler so that all diagnostics are visible during `west build`. Without this,
any construct classified as a `diagnostic` (warning) instead of an `error` is
invisible to the developer.

This is the single highest-impact fix for error message quality. Every
unsupported construct that produces invalid C should either:
1. Be classified as `module.errors` (shown regardless of flags), OR
2. `--strict` should be passed by CMake so `module.diagnostics` are treated as errors

Recommended fix: add `--strict` to both the configure-time `execute_process` and
the build-time shell script in `oz_transpile.cmake`.

## Workaround

None practical â€” the user would have to manually edit oz_transpile.cmake or run
the transpiler outside CMake with `--verbose`.
