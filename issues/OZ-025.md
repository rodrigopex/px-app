## Context

Phase 2 mutation testing, Step 2. Testing error message quality for documented
unsupported construct: boxed expressions `@(expr)`.

## Input

```objc
/* In PXMovingAverageFilter.m, -processValue:fromSensor: method */
OZNumber *r = @(_sum / _count);
return [r intValue];
```

## Observed

Transpiler exits successfully (no error, no diagnostic visible). Generated C:

```c
struct OZNumber * r = /* TODO: ObjCBoxedExpr(ParenExpr) */;
```

GCC then fails:
```
PXMovingAverageFilter_ozm.c:51:67: error: expected expression before ';' token
   51 |         struct OZNumber * r = /* TODO: ObjCBoxedExpr(ParenExpr) */;
```

- No transpiler error
- No transpiler diagnostic/warning
- Generated C is invalid (bare comment where an expression should be)
- GCC error points at generated C file, not source .m file
- Developer must know transpiler internals to understand "ObjCBoxedExpr"

## Expected

Transpiler should emit an error:
```
oz_transpile: error: PXMovingAverageFilter.m:23: boxed expression @(...) is not supported
  Use OZNumber literals or explicit [[OZNumber alloc] initWithInt:] instead
```

The construct should be classified as `module.errors` since it produces invalid C.
A TODO comment in generated output is never acceptable — it always causes a
downstream GCC failure.

## Workaround

Use `[[OZNumber alloc] initWithInt:value]` instead of `@(value)`, or avoid
OZNumber boxing entirely and use plain `int`.
