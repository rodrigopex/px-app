## Context

Calling a dispatched method on an ivar inside a short method body. The transpiler
emits `OZ_SEND_count(_oz_recv0)` but never declares `_oz_recv0`.

## Input

```objc
@interface PXSensorRegistry : OZObject {
    OZArray *_sensors;
}
- (int)sensorCount;
@end

@implementation PXSensorRegistry
- (int)sensorCount {
    return [_sensors count];
}
@end
```

## Observed

Generated C:
```c
int PXSensorRegistry_sensorCount(struct PXSensorRegistry *self)
{
    return OZ_SEND_count(_oz_recv0);
}
```

GCC error:
```
error: '_oz_recv0' undeclared (first use in this function)
```

The receiver variable declaration (`struct OZObject *_oz_recv0 = (struct OZObject *)self->_sensors;`)
is missing. Other methods in the same class (e.g., `readAll`) correctly emit
receiver variables for dispatch calls on ivars.

## Expected

```c
int PXSensorRegistry_sensorCount(struct PXSensorRegistry *self)
{
    struct OZObject *_oz_recv0 = (struct OZObject *)self->_sensors;
    return OZ_SEND_count(_oz_recv0);
}
```

## Workaround

Avoid dispatch by storing the ivar in a local variable first, or restructure
to avoid calling dispatched methods directly in a return statement.
See WORKAROUNDS.md entry WA-003.

## Discussion

The `readAll` method in the same class correctly emits receiver variables for
dispatch calls on ivars. The difference might be that `sensorCount` uses the
dispatch call in a `return` statement, while `readAll` uses it in a regular
expression statement.

## Resolution

Fixed in `emit.py`: `_emit_return_stmt` now buffers the return expression before
writing it, so any `pre_stmts` generated by `_emit_expr` (e.g., protocol dispatch
receiver variable `_oz_recv0`) are flushed before the `return` keyword.

Test added: `test_return_protocol_dispatch_emits_receiver_var` — verifies the
receiver variable declaration appears before the return statement.
